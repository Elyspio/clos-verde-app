Configuration API des notifications Web Push

Cette page décrit la configuration côté API pour autoriser les notifications navigateur quand un utilisateur est mentionné dans un sujet non muet ou quand une nouvelle réservation est créée.

Principe

Le navigateur demande la permission via requestNotificationPermission(). Si l'utilisateur accepte, le frontend crée un abonnement Push API et l'enregistre auprès de l'API. L'API conserve cet abonnement par utilisateur et envoie ensuite les notifications via Web Push, même quand le site n'est pas ouvert.

Variables API requises

La section de configuration s'appelle PushNotifications. En développement, elle peut rester désactivée. En production, les trois valeurs VAPID doivent être définies.

Principe de fonctionnement de VAPID

VAPID signifie Voluntary Application Server Identification. C'est le mécanisme standard qui permet à l'API Clos Verde de s'identifier auprès des services push des navigateurs (Chrome/Edge, Firefox, Safari compatible Web Push) sans créer un compte fournisseur par navigateur.

Le système repose sur une paire de clés asymétriques :

  • la clé publique est envoyée au frontend via /api/notifications/vapid-public-key ;
  • la clé privée reste uniquement côté API et sert à signer les envois Web Push.

Lorsque l'utilisateur clique sur Activer les notifications, le frontend demande d'abord la permission navigateur. Si elle est accordée, il récupère la clé publique VAPID de l'API et appelle registration.pushManager.subscribe(...). Le navigateur crée alors un abonnement contenant :

  • un endpoint, c'est-à-dire l'URL du service push du navigateur ;
  • une clé p256dh, utilisée pour chiffrer le contenu envoyé ;
  • un secret auth, utilisé par le protocole Web Push.

Cet abonnement est enregistré par l'API pour l'utilisateur connecté. Quand un événement doit notifier l'utilisateur, l'API construit un petit payload JSON puis contacte l'endpoint du navigateur. La librairie Web Push signe la requête avec la clé privée VAPID et ajoute un JWT d'identification contenant notamment le Subject. Le service push vérifie cette signature avec la clé publique associée à l'abonnement.

La clé privée VAPID n'est pas une clé de permission utilisateur. Elle prouve seulement que l'API qui envoie la notification est bien le serveur applicatif qui a demandé l'abonnement. La permission utilisateur reste gérée par le navigateur : si l'utilisateur refuse ou révoque la permission, l'API ne peut pas forcer l'affichage d'une notification.

Le contenu de notification transite par le service push du navigateur, mais il est chiffré avec les informations de l'abonnement. L'API ne doit tout de même envoyer que des données courtes et non sensibles : titre, extrait, URL relative et tag de notification.

Clé Description
PushNotifications:Enabled true pour activer l'exposition de la clé publique et l'envoi Web Push.
PushNotifications:Subject Contact VAPID. Utiliser une URL ou une adresse email, par exemple mailto:admin@example.com.
PushNotifications:PublicKey Clé publique VAPID retournée au frontend.
PushNotifications:PrivateKey Clé privée VAPID utilisée uniquement par l'API. Ne jamais l'exposer au frontend.

Exemple appsettings

{
  "PushNotifications": {
    "Enabled": true,
    "Subject": "mailto:admin@example.com",
    "PublicKey": "B...",
    "PrivateKey": "..."
  }
}

Exemple variables d'environnement

En conteneur ou en hébergement cloud, préférer les variables d'environnement afin de ne pas commiter la clé privée. 

PushNotifications__Enabled=true
PushNotifications__Subject=mailto:admin@example.com
PushNotifications__PublicKey=B...
PushNotifications__PrivateKey=...

Endpoints utilisés par le frontend

Méthode Route Rôle
GET /api/notifications/vapid-public-key Retourne la clé publique VAPID si les notifications sont activées.
PUT /api/notifications/push-subscription Enregistre ou remplace l'abonnement Push API du navigateur pour l'utilisateur connecté.
DELETE /api/notifications/push-subscription Supprime l'abonnement du navigateur pour l'utilisateur connecté.

Générer des clés VAPID

Utiliser un générateur VAPID compatible Web Push. Exemple avec l'outil Node web-push : 

npx web-push generate-vapid-keys

Copier la clé publique dans PushNotifications:PublicKey et la clé privée dans PushNotifications:PrivateKey.

Checklist de déploiement

  • Servir le frontend en HTTPS. Les permissions navigateur et les service workers exigent un contexte sécurisé.
  • Configurer PushNotifications:Enabled=true sur l'API.
  • Configurer Subject, PublicKey et PrivateKey.
  • Vérifier que l'utilisateur est authentifié avant d'appeler les endpoints /api/notifications/*.
  • Vérifier que le service worker /push-worker.js est servi par le frontend.
  • Tester avec un message mentionnant l'utilisateur dans un topic non muet.
  • Tester avec une nouvelle réservation créée par un autre utilisateur.

Si la configuration VAPID est absente ou désactivée, l'API ne fournit pas de clé publique et le frontend ne peut pas créer d'abonnement Web Push. Les notifications en arrière-plan sont alors indisponibles.
